# 民事纠纷案件诉状生成与法律咨询系统

这是一个面向中国大陆小额、事实清楚、证据相对明确的民事纠纷场景的轻量级法律辅助 Demo。系统需要重点实现“法律条文与起诉流程咨询”，并在此基础上提供“民事起诉状生成”能力。

系统主要通过本地法律知识库检索与大模型生成，为用户解答民事起诉相关问题，并根据用户填写的案件信息生成结构完整、要素清晰的民事起诉状初稿。页面中的所有文字说明必须使用中文，项目中的提示词也必须使用中文。

这是一个全栈工程师的轻量级开发项目，需要包括下面几个部分：

* 前端页面
* 后端服务
* 测试用例

项目体量必须严格控制，以 1 名开发人员在 1 天内完成可运行 Demo 为目标。优先保证核心流程完整、回答可追溯、页面易用，不开发复杂后台和非必要功能。

## 项目边界

### 适用场景

* 案件标的较小、事实关系相对简单、证据材料相对清晰的民事纠纷。
* 默认支持以下高频案件类型：
  * 民间借贷纠纷
  * 房屋租赁合同纠纷
  * 买卖合同纠纷
* 用户可以咨询法律条文、起诉条件、管辖法院、立案流程、材料准备、诉讼费用、证据整理等问题。
* 用户可以通过表单或自然语言描述案件，生成民事起诉状初稿。

### 不适用场景

* 刑事案件、行政案件、婚姻家庭重大争议、劳动争议仲裁前置等复杂程序。
* 当事人人数众多、证据关系复杂、需要专业律师判断的案件。
* 需要联网核验最新法规、在线立案、电子签章、律师审核、案件胜算预测等功能。
* 系统输出仅为法律信息参考和文书初稿，不构成正式法律意见，页面与接口结果中必须保留明确提示。

## 前端页面

* 前端页面使用 Vue 3 开发框架，代码要求规范清晰。
* 页面设计应当简约大气、专业可信，适当增加轻量动效，不使用过度年轻化或娱乐化的视觉风格。
* 前端页面包括以下页面，每个页面都有详细说明，请仔细阅读并完成开发。

### 首页

* 简要说明系统能力、适用案件类型和输出限制。
* 提供两个核心入口：
  * 法律咨询
  * 起诉状生成
* 展示系统免责声明：系统仅提供法律信息参考，不能替代律师的专业法律意见。

### 法律咨询页面

法律咨询页面是项目的核心页面，必须优先完成。

* 支持用户选择咨询类型，包括：
  * 法律条文咨询
  * 起诉条件咨询
  * 管辖法院咨询
  * 起诉流程咨询
  * 材料与证据咨询
  * 诉讼费用咨询
* 支持用户以自然语言输入问题。
* 系统根据用户问题检索本地法律知识库，并结合大模型生成中文回答。
* 回答必须采用清晰结构，默认包含以下部分：
  * 简要结论
  * 适用条文
  * 起诉流程或处理步骤
  * 需要准备的材料
  * 风险提示
  * 建议补充的信息
* 回答中涉及法律条文时，必须标注条文名称和条号，例如《中华人民共和国民事诉讼法》第一百二十二条。
* 回答下方展示引用来源，包括文档名称、条文号或段落标题，便于用户追溯答案依据。
* 当知识库中没有足够依据时，不能编造条文，应明确提示“当前知识库未检索到充分依据”，并建议用户补充问题或咨询专业律师。
* 当用户描述不完整时，系统应提出最多 3 个关键追问，例如被告身份、合同履行地、借款金额、证据类型等。
* 支持流式输出，提升问答体验。
* 支持复制回答内容。
* 支持保留当前浏览器中的最近咨询记录，刷新页面后仍可查看；不需要开发复杂的多用户历史记录后台。

### 起诉状生成页面

* 提供结构化表单，帮助用户填写民事起诉状所需信息。
* 表单至少包括以下字段：
  * 案件类型
  * 原告姓名、证件号码、联系方式、住址
  * 被告姓名或名称、证件号码或统一社会信用代码、联系方式、住址
  * 诉讼请求
  * 事实与理由
  * 证据清单
  * 拟提交法院
  * 起诉日期
* 支持用户输入一段自然语言案件描述，系统自动提取并填充上述表单字段。
* 对必填字段进行校验，缺少关键信息时给出明确提示。
* 对无法确认的信息，不允许自行编造，应在文书中以“【待补充】”标记。
* 根据选择的案件类型使用对应文书模板生成民事起诉状。
* 生成的起诉状至少包含：
  * 文书标题
  * 原告信息
  * 被告信息
  * 诉讼请求
  * 事实与理由
  * 证据和证据来源
  * 此致法院
  * 具状人
  * 日期
* 支持在线预览、复制和下载。下载格式至少支持 Markdown 或 TXT，时间允许时可支持 DOCX。
* 生成结果下方必须展示“请核对事实与金额，正式提交前建议由专业律师审核”的提示。

### 法律依据页面

* 展示当前系统内置的法律知识来源。
* 每条知识来源至少展示：
  * 文件名称
  * 法律或流程类别
  * 适用说明
  * 当前索引状态
* 允许用户查看检索到的条文片段。
* Demo 阶段不需要开发完整的知识库上传、解析和管理后台。

## 后端服务

* 后端使用 FastAPI 框架开发，代码要求规范清晰。
* Demo 阶段使用 SQLite 保存咨询记录和生成记录，不引入 MySQL，降低部署成本。
* 向量检索优先使用 FAISS 或 Chroma 等本地轻量级方案，不要求安装 Milvus。
* 大模型默认使用通义千问模型，模型名称默认为 qwen-max；请创建好配置文件，用户可自行修改 API Key、模型名称和服务地址。
* 重要配置统一放入配置文件，包括：
  * 大模型 API Key
  * 模型名称
  * 温度、topP、最长输出 token 数
  * 检索 topK
  * 法律知识库目录
  * 数据库路径
* 大模型回答必须支持流式输出。
* 如果未配置大模型 API Key，系统可以返回可读的规则模板结果，并在结果中明确标注“当前为离线演示结果”。

### 法律知识库

* 法律知识文件统一存放在项目 `data/legal` 目录中。
* 至少内置以下类型的本地知识文件：
  * 民事法律基础条文摘录
  * 民事诉讼起诉条件与程序条文摘录
  * 立案材料与起诉流程说明
  * 诉讼费用基础说明
  * 民间借贷、房屋租赁、买卖合同三类案件提示
* 法律条文切块时应优先按“条”切分，避免将同一条文拆散。
* 每个知识块必须保留来源文件、标题、条文号等元数据。
* 检索默认返回 topK=5 的上下文，并在回答中展示引用来源。
* 法律咨询提示词必须要求大模型：
  * 优先依据检索到的知识库内容回答
  * 不编造法律条文、条号、流程和费用标准
  * 信息不足时明确说明并追问
  * 使用简洁、普通用户容易理解的中文
  * 保留必要的法律风险提示

### 核心接口

至少提供以下后端接口：

* 咨询问答接口：接收用户问题、咨询类型和历史对话，返回流式回答及引用来源。
* 案件信息提取接口：接收自然语言案件描述，返回结构化表单字段。
* 起诉状生成接口：接收结构化案件信息，返回民事起诉状初稿。
* 法律依据列表接口：返回当前知识库文件和索引状态。
* 健康检查接口：返回服务运行状态。

## 测试用例

需要严格遵守开发惯例，对重要功能生成测试用例。测试用例至少包括：

* 检查法律咨询接口是否能正常返回结果。
* 检查法律条文咨询结果是否包含条文名称、条号和引用来源。
* 检查起诉流程咨询结果是否包含明确的流程步骤。
* 检查知识库依据不足时，系统是否会明确提示，而不是编造答案。
* 检查起诉状生成接口在缺少必填字段时是否能返回明确错误。
* 检查起诉状中的未知信息是否以“【待补充】”标记。
* 检查民间借贷、房屋租赁、买卖合同三类案件是否能正常生成对应起诉状。
* 检查前端咨询页面和起诉状生成页面的核心交互流程是否可用。
* 每次进行功能修改以后，都要运行测试用例。

## 一天开发优先级

为了保证 1 天内完成 Demo，必须按照以下优先级开发：

1. 本地法律知识库整理与检索
2. 法律咨询页面和流式问答接口
3. 条文来源引用展示
4. 起诉状结构化表单
5. 起诉状模板生成与下载
6. 自然语言案件描述自动提取
7. 法律依据页面
8. 历史记录与页面动效优化

前 5 项为必须完成内容，后 3 项在时间允许时完成。
